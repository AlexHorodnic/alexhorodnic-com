Product Context

The problem

Operations teams often work across account records, implementation queues, activity feeds, and reporting tools. The project explores how those workflows can live in one coherent interface without turning the product into a collection of disconnected admin screens.

I designed and built the complete frontend, including the information architecture, interaction states, responsive behavior, reusable components, and realistic demo data.

Engineering Approach

Architecture that stays readable

The codebase uses a feature-first Angular structure with clear boundaries between route features, shared UI, core models and services, and mock data.

01

Feature-owned UI

Overview, accounts, workflow, and analytics own their page behavior while reusable drawers, badges, KPI cards, and empty states remain shared.

02

Predictable state

Angular signals and computed values manage local view state. RxJS models delayed data flows and cleanup uses Angular's lifecycle interop.

03

Deliberate styling

SCSS is split into base, layout, component, page, and responsive layers so global design decisions do not leak into feature logic.

Key Challenge

One workflow, two interaction models

A desktop Kanban board benefits from direct manipulation, but dragging cards becomes awkward and error-prone on narrow touch screens.

The decision

Desktop users get Angular CDK drag-and-drop with visible drop targets and custom drag previews. On smaller screens, dragging is disabled and the board becomes a stacked task flow with explicit movement controls.

This trades interaction consistency for usability: the workflow remains the same, while the control mechanism matches the device.

Engineering Quality

Designed beyond the happy path

Responsive behavior

Tables become cards, navigation becomes a sheet, and drawers adapt to mobile sizing.

Complete UI states

Core pages include loading, error, empty, filtered, selection, and feedback states.

Accessible interaction

Controls include focus states, labels, keyboard behavior, and touch-friendly targets.

Operational details

CSV export, bulk actions, task comments, activity history, and undoable deletion add realism.

Tradeoffs

What this project is and is not

This is a frontend case study, not a production SaaS product. Data is mocked and state is managed in the browser; there is no authentication, backend, or customer data.

Automated tests now cover the highest-value data, filtering, workflow, analytics, export, and route-loading behavior. A real product would still require API contracts, authentication, permissions, server persistence, observability, and server-side validation.
